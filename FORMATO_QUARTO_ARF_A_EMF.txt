GUIA DE MIGRACION DE FORMATO: tutorial_arf -> tutorial_emf
===========================================================

Objetivo del documento
----------------------

Este documento describe como llevar el formato actual de:

  C:\Users\DELL\Desktop\MartinGit\tutorial_arf

hacia:

  C:\Users\DELL\Desktop\MartinGit\tutorial_emf

sin copiar el contenido sustantivo del libro. La migracion debe copiar la
arquitectura, la navegacion, el estilo visual, los parciales HTML, el flujo de
publicacion y la organizacion Quarto. El contenido financiero, los ejemplos,
las figuras, los datos y los textos de EMF deben seguir siendo de EMF.

Principio rector:

  Todo debe moverse a Quarto. El resultado final debe ser un Quarto book,
  renderizado a _book, publicado por GitHub Actions / GitHub Pages, sin depender
  de HTML generado en la raiz del repositorio.

Paquete portable:

  Este documento debe viajar junto con la carpeta:

    quarto_format_template/

  Si el nuevo proyecto tiene este .txt y esa carpeta, Codex no necesita buscar
  tutorial_arf. Las instrucciones y los archivos base estan contenidos aqui.
  tutorial_arf solo fue la fuente original del formato.

Instruccion obligatoria de trabajo:

  - Trabajar directamente en main.
  - No crear una rama nueva para esta migracion.
  - No crear una rama _book.
  - No usar _book como rama ni como carpeta versionada.
  - _book es output generado por quarto render.
  - La publicacion debe hacerse con GitHub Actions / GitHub Pages.
  - Si Codex esta en tutorial_emf, debe usar este documento y
    quarto_format_template/ como fuente del formato.


1. Diagnostico actual
---------------------

tutorial_arf ya esta organizado como libro Quarto moderno:

  - Tiene project: type: book en _quarto.yml.
  - Renderiza todo dentro de _book.
  - _book es output generado y esta ignorado por git.
  - La publicacion se hace desde main mediante GitHub Actions.
  - GitHub Pages debe usar "GitHub Actions" como fuente de Pages.
  - No se usa una segunda rama _book.
  - La navegacion lateral es de book, no un TOC simple de documento.
  - La pagina inicial es index.qmd con Preface y What's new.
  - El sitio tiene CSS propio en styles.css.
  - El boton de subir, la barra de progreso, el menu lateral, la navegacion
    local y el menu "On this page" se inyectan con parciales HTML.

tutorial_emf todavia esta en un formato mas antiguo:

  - No tiene project: type: book en _quarto.yml.
  - Tiene index.html generado en la raiz.
  - Tiene carpetas generadas como index_files y robotrader_files.
  - Sus fuentes principales siguen siendo .rmd:
      ffpp3.rmd
      fh2o.rmd
      robotrader.rmd
      robotrader - copia.rmd
  - Solo tiene index.qmd como archivo Quarto principal.
  - Usa toc-location: left como documento largo, no como book.
  - No tiene styles.css.
  - back-to-top.html y progressbar.html son versiones viejas.
  - include-in-header incluye parciales que deberian ir como fragmentos limpios,
    no como documentos HTML completos.

La migracion de EMF debe convertir el proyecto a book Quarto, copiar el molde
de ARF y eliminar la dependencia de artefactos HTML generados en la raiz.

  {{< include nombre-del-fragmento.qmd >}}

dentro de un capitulo .qmd, como se hace en tutorial_arf.


3. Regla de publicacion y ramas
-------------------------------

No crear una rama adicional para _book.

Flujo correcto:

  1. Editar fuentes en main.
  2. Renderizar localmente cuando convenga.
  3. Commit en main.
  4. Push a origin/main.
  5. GitHub Actions ejecuta quarto render.
  6. GitHub Actions sube _book como artefacto de Pages.
  7. GitHub Pages publica ese artefacto.

_book no debe versionarse. Es output generado.

En GitHub, la configuracion de Pages debe quedar como:

  Source: GitHub Actions

No usar:

  - gh-pages
  - una rama _book
  - root /docs como output manual
  - index.html generado en la raiz

5. _quarto.yml objetivo
-----------------------

El _quarto.yml de EMF debe transformarse de documento HTML largo a book.
Usar quarto_format_template/_quarto.yml.template como punto de partida y adaptar
titulo, footer, analytics y capitulos:

project:
  type: book
  output-dir: _book

book:
  title: "Forecasting with R"
  subtitle: "Book"
  author: "Dr. Martin Lozano"
  page-navigation: true
  search: true
  date: today
  date-format: "MMMM D, YYYY"
  sidebar:
    style: docked
    collapse-level: 1
  page-footer:
    left: "Forecasting with R"
    right: 'Dr. Martin Lozano &middot; <a href="mailto:mlozanoqf@gmail.com">mlozanoqf@gmail.com</a> &middot; <a href="https://mlozanoqf.github.io/">mlozanoqf.github.io</a>'
  chapters:
    - text: "Forecasting with R"
      href: index.qmd
    - 01-[primer-capitulo].qmd
    - 02-[segundo-capitulo].qmd
    - 03-[tercer-capitulo].qmd
    - references.qmd

bibliography: references.bib
link-citations: true

crossref:
  chapters: true

execute:
  warning: false
  message: false

format:
  html:
    theme:
      light: flatly
      dark: darkly
    toc: false
    toc-title: "On this page"
    toc-location: right
    toc-depth: 2
    smooth-scroll: true
    code-fold: true
    code-copy: true
    code-tools:
      toggle: true
      source: false
    code-block-bg: true
    code-block-border-left: "#31BAE9"
    code-line-numbers: true
    number-sections: true
    number-depth: 3
    page-layout: full
    include-after-body:
      - analytics_emf.html
      - back-to-top.html
      - progressbar.html
      - sidebar-toggle.html
      - local-navigation.html
      - page-sections.html
      - sidebar-accordion.html
      - sidebar-chapter-sections.html
    css: styles.css

Notas:

  - Cambiar analytics_emf.html por el nombre correcto del parcial de EMF.
  - No usar include-in-header para back-to-top ni progressbar.
  - No poner HTML completo con <html>, <head> o <body> en los parciales.
  - No usar toc-location: left como navegacion principal. La navegacion principal
    debe ser la sidebar del book.
  - toc: false es intencional. El menu "On this page" lo crea page-sections.html.

problema resuelve y como conecta intuicion, modelos, datos, graficas y codigo.
No debe repetir el titulo del libro, porque Quarto ya muestra el titulo.
:::

## What's new in this edition {.unnumbered}

- **Cambio sustantivo 1:** descripcion corta, orientada al lector.
- **Cambio sustantivo 2:** descripcion corta, orientada al lector.
- **Cambio sustantivo 3:** descripcion corta, orientada al lector.

Reglas de la pagina inicial:

  - No repetir el titulo del libro encima de Preface.
  - Preface debe aparecer justo despues del bloque de titulo generado por Quarto.
  - What's new debe ser lista bullet sintetica, no tarjetas.
  - Las bullets deben hablar de cambios relevantes para el lector, no de detalles
    internos como "se arreglo un CSS".
  - El bloque de contacto aparece arriba mediante contact.html.

7. Organizacion de capitulos
----------------------------

Cada capitulo debe ser un archivo .qmd listado en _quarto.yml.

Patron:

  01-[tema].qmd
  02-[tema].qmd
  03-[tema].qmd

Dentro de cada capitulo:

  # Titulo del capitulo

  Introduccion narrativa del capitulo.

  {{< include 01-fragmento-a.qmd >}}

  {{< include 01-fragmento-b.qmd >}}

  {{< include 01-fragmento-c.qmd >}}

Los fragmentos tambien deben ser .qmd.

Reglas:

  - Evitar que index.qmd sea el archivo que carga todo el contenido.
  - Evitar child chunks antiguos de R Markdown si se esta migrando a Quarto.
  - Evitar archivos .rmd en el flujo final.
  - Mantener los nombres de archivo con prefijos numericos para que el orden sea
    claro.
  - Usar un solo # para el titulo del capitulo.
  - Usar ## para las secciones principales del capitulo.
  - Evitar ### si no se quiere un tercer nivel visible o confuso.
  - Para subtitulos internos que no deben entrar como secciones numeradas, usar
    un bloque visual tipo:

      ::: {.book-subhead}
      Texto del subtitulo interno
      :::

    En ARF existe .merton-subhead para ese uso. Para EMF conviene crear una
    clase generica .book-subhead en styles.css o renombrar .merton-subhead.


8. Navegacion lateral
---------------------

La navegacion lateral viene de Quarto book y se mejora con tres parciales:

  sidebar-toggle.html
  sidebar-accordion.html
  sidebar-chapter-sections.html

Comportamiento objetivo:

  - Sidebar docked a la izquierda.
  - El link de la pagina inicial no debe aparecer duplicado dentro de la lista.
  - Cada capitulo puede desplegar sus secciones principales.
  - Solo un bloque de sidebar debe quedar abierto a la vez.
  - El capitulo activo debe resaltarse.
  - La seccion activa dentro del capitulo debe resaltarse.
  - El boton de sidebar permite ocultar/mostrar la navegacion.
  - Si la sidebar se oculta, el contenido debe usar mas ancho disponible.

El archivo que mas requiere adaptacion manual es:

  sidebar-chapter-sections.html

En ese archivo hay un arreglo chapters. Para EMF debe quedar con las paginas y
secciones reales de EMF. Ejemplo de estructura:

  var chapters = [
    {
      page: "01-introduction.html",
      sections: [
        { id: "forecasting-problem", text: "1.1 The forecasting problem" },
        { id: "data-and-baseline-models", text: "1.2 Data and baseline models" }
      ]
    },
    {
      page: "02-machine-learning.html",
      sections: [
        { id: "model-workflow", text: "2.1 Model workflow" },
        { id: "accuracy-and-loss-functions", text: "2.2 Accuracy and loss functions" }
      ]
    }
  ];

Como obtener los ids:

  1. Renderizar el capitulo.
  2. Abrir el HTML generado en _book.
  3. Buscar los section id generados por Quarto.
  4. Copiar esos ids en sidebar-chapter-sections.html.
  5. Volver a renderizar y verificar que los links bajan exactamente a la
     seccion correcta.

No dejar nombres de secciones de ARF dentro de EMF.


9. Navegacion local "On this page"
----------------------------------

ARF ya no depende del TOC lateral nativo de Quarto para cada pagina. El archivo
page-sections.html crea una barra local superior con:

  - breadcrumbs,
  - un boton "On this page",
  - dropdown con las secciones de la pagina actual.

Comportamiento objetivo:

  - No aparece en la pagina inicial.
  - Aparece en paginas de capitulos.
  - Se mantiene sticky arriba.
  - Lee h2 y h3, pero conviene que el libro use principalmente h2.
  - Al hacer click, hace smooth scroll y corrige el offset para que el titulo no
    quede tapado.
  - En movil no debe salirse del ancho de la pantalla.

CSS relevante:

  .page-local-nav
  .page-section-menu
  .page-section-toggle
  .page-section-list
  .page-section-link
  .page-section-link-h3

11. Barra de progreso
---------------------

Archivo:

  progressbar.html

Comportamiento ARF:

13. CSS global
--------------

Copiar styles.css de ARF.

El CSS define variables para tema claro y oscuro:

  body.quarto-light:
    --book-link
    --book-link-hover
    --book-active-link
    --book-backtop-bg
    --book-backtop-hover
    --book-rule
    --book-surface
    --book-soft-surface
    --book-card-border
    --book-muted
    --book-accent
    --book-accent-soft

  body.quarto-dark:
    mismas variables adaptadas a dark mode

Aspecto general:

  - Links azules.
  - Link activo en tono naranja/cobre.
  - Superficies blancas o suaves en light mode.
  - Superficies oscuras en dark mode.
  - Bordes discretos.
  - No usar paleta monotona.
  - No usar tarjetas decorativas para todo.
  - Radio de bordes moderado.

Reglas de layout importantes:

  - html scroll-padding-top: 4.5rem.
  - main [id] scroll-margin-top: 4.5rem.
  - tablas al 100% de ancho.
  - .cell-output-display con overflow-x auto.
  - ecuaciones MathJax display con overflow-x auto.
  - breadcrumbs con ellipsis para no romper el ancho.
  - footer en una linea cuando hay espacio y responsive en movil.

Clases especificas de contenido:

  styles.css de ARF incluye clases como .merton-state-table y .merton-subhead.
  Para EMF hay dos opciones:

  - Copiar styles.css completo y dejar esas clases aunque no se usen.
  - Limpiar o renombrar clases especificas si se quiere que el CSS sea mas
    generico.

Recomendacion para EMF:

  - Mantener las clases globales.
  - Crear .book-subhead como clase generica si EMF necesita subtitulos internos
    no numerados.
  - Evitar nombres como .merton-subhead en EMF salvo que no moleste que queden
    como clase tecnica invisible.

Ejemplo de clase generica:

  .book-subhead {
    color: inherit;
    font-size: 1.08rem;
    font-weight: 700;
    margin: 1.35rem 0 0.45rem;
  }

  .book-subhead > p {
    margin: 0;
  }

17. Figuras, tablas y ancho de pantalla
---------------------------------------

Reglas de formato heredadas de ARF:

  - Evitar elementos que ensanchen la pantalla.
  - Tablas al 100% con overflow horizontal si hace falta.
  - Si una tabla es muy ancha, abreviar nombres de columnas antes de aceptar
    overflow excesivo.
  - Ecuaciones display deben poder hacer scroll horizontal.
  - Captions deben ser claros y no demasiado largos.
  - Usar subindices con math, por ejemplo $V_T$, no texto plano V_T en captions.
  - En figuras ggplot, preferir etiquetas parseables o expresiones cuando se
    necesiten subindices.
  - No dejar leyendas que tapen datos importantes.
  - Revisar figuras en HTML despues de renderizar, no solo en el codigo.

Aunque estas reglas nacieron en ARF, son parte del formato editorial que conviene
llevar a EMF.


18. GitHub Actions para publicar
--------------------------------

Copiar quarto_format_template/.github/workflows/publish.yml como base.

Mantener:

  name: Publish Quarto book

  on:
    push:
      branches:
        - main
    workflow_dispatch:

  permissions:
    contents: read
    pages: write
    id-token: write

  jobs:
    build:
      runs-on: windows-2022

Puntos importantes:

  - Usar actions/checkout con fetch-depth: 0.
  - Instalar Quarto con Chocolatey.
  - Instalar R con Chocolatey.
  - Pinnear R a 4.5.2 si se quiere replicar el flujo estable de ARF.
  - Ejecutar quarto render.
  - Subir _book con actions/upload-pages-artifact.
  - Desplegar con actions/deploy-pages.

La lista de paquetes R debe adaptarse a EMF. Revisar el codigo EMF y listar
solo los paquetes necesarios para renderizar. La plantilla portable incluye una
lista inicial con paquetes comunes para EMF, como fontawesome, tictoc, fpp3 y
h2o, pero esa lista debe verificarse contra el contenido real.

Ejemplo de bloque que debe adaptarse:

  install.packages(c(
    "dplyr",
    "ggplot2",
    "knitr",
    "rmarkdown"
    # agregar aqui paquetes especificos de EMF
  ))

Si EMF usa paquetes pesados como h2o, fpp3, modeltime, tidymodels, forecast,
tsibble, fable, lubridate, timetk, etc., agregarlos aqui segun el codigo real.

24. Decisiones editoriales que deben conservarse
------------------------------------------------

Estas decisiones forman parte del formato, aunque no son puramente tecnicas:

  - Llamarlo "book", no "tutorial", en el texto visible.
  - Evitar una landing page tipo marketing. La pagina inicial debe orientar al
    lector y llevarlo al contenido.
  - Mantener prosa explicativa, no solo codigo.
  - Conectar ecuaciones, codigo, tablas, figuras e interpretacion.
  - Evitar abstracciones que no se demuestran.
  - Evitar conclusiones demasiado breves debajo de figuras importantes.
  - Preferir bullets sinteticas para cambios de edicion.
  - Evitar tarjetas decorativas para listas de cambios.
  - Evitar exceso de dos puntos en titulos y prosa.
  - Evitar el estilo repetido de negar una interpretacion y luego corregirla.
    Mejor afirmar directamente la logica correcta.
  - Usar valores calculados por codigo cuando sean resultados, no numeros
    escritos a mano en parrafos.
  - Revisar el HTML renderizado, no solo el qmd.


25. Nota sobre Book edition y Publication
-----------------------------------------

ARF usa R/book-edition.R para mostrar:

  - Book edition: hash corto
  - Publication: N
  - First publication: fecha

Publication se calcula con:

  git rev-list --count HEAD

Por eso el workflow necesita:

  fetch-depth: 0

El hash corto se muestra en el subtitulo. En el ARF actual se calcula a partir
de los .qmd detectados en _quarto.yml. Si EMF usa muchos fragmentos incluidos
desde capitulos, conviene revisar despues si el hash debe incluir tambien esos
fragmentos. Para publicacion practica, el numero Publication ya cambia con cada
commit.
